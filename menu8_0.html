Factory Method Pattern

"객체 생성을 위한 인터페이스를 정의하지만, 인스턴스를 생성할 클래스는 서브클래스가 결정하도록 한다. 팩터리 메서드는 객체 생성을 서브클래스에게 맡긴다"

설명
객체를 생성하는 책임을 상위 클래스가 아니라 하위 클래스에게 넘기는 패턴입니다.
즉, 상위 클래스는 '어떤 종류의 객체를 만들지'에 대한 결정을 하지 않고,
하위 클래스가 구체적인 객체를 만들어내는 역할을 합니다.

🔵 팩터리 메서드 패턴을 사용하는 이유

이유 설명
객체 생성 코드를 분리 객체 생성 로직을 클라이언트 코드로부터 분리하여, 코드의 의존성을 낮출 수 있다.
변경에 유연함 어떤 객체를 생성할지 결정하는 책임을 서브클래스에 맡겨서, 새 클래스를 추가해도 기존 코드를 수정할 필요가 거의 없더.
코드 재사용성 증가 상위 클래스에서는 제품 생성 방식을 일반화할 수 있고, 하위 클래스는 필요한 부분만 오버라이드해서 사용할 수 있다.
객체 생성을 캡슐화 객체 생성 방식을 숨길 수 있어, 클라이언트는 구체적인 클래스를 몰라도 객체를 사용할 수 있다.
CP(개방-폐쇄 원칙) 준수 새로운 객체 타입을 추가할 때 기존 코드를 수정하지 않고 확장할 수 있다.

🟠 팩터리 메서드 패턴의 활용 사례

활용 분야 구체적 설명
프레임워크 개발 Spring Framework: ApplicationContext가 Bean 생성을 내부적으로 팩터리 메서드 패턴으로 처리
GUI 라이브러리 다양한 버튼(Button)이나 윈도우(Window)를 운영체제별로 생성하는 경우 (WindowsButton, MacButton 등)
데이터베이스 연결 다양한 DB 연결 객체 (MySQLConnection, OracleConnection)를 팩터리 메서드를 통해 생성
문서 편집기 프로그램 새로운 문서(Document)나 뷰(View)를 생성할 때, 팩터리 메서드를 이용해서 다양한 종류를 지원
게임 개발 다양한 몬스터(Monster), 무기(Weapon) 객체를 팩터리 메서드를 통해 생성하여 확장성 확보
로깅 라이브러리 다양한 로그 저장 방식 (ConsoleLogger, FileLogger, DatabaseLogger 등)을 팩터리 메서드로 생성
