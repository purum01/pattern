✅ 팩터리 패턴을 고려해야 하는 경우

객체 생성 과정이 복잡하거나, 객체 생성 프로세스를 캡슐화하고 싶을 때 적용할 수 있습니다.

주요 상황

1. 객체 설정 복잡성/유사성 문제

  • 규칙 설정 클래스처럼 객체 설정 코드가 복잡하거나 유사한 경우 → 팩터리 패턴 사용.
  • 클래스마다 객체 설정 코드가 복잡하면 유지보수가 어려워짐.
  • 팩터리 메서드 패턴을 통해 객체 생성/설정 분리 → 클래스 복잡도 감소.

2. 생성할 객체가 상황(흐름)에 따라 달라질 때

  • 상황(흐름)에 따라 생성할 객체가 달라질 경우.
  • 클래스 내부 if-else/switch를 없애고, 팩터리 클래스를 통해 관리.

✨ 팩터리 패턴의 4가지 역할

번호 설명
1 팩터리 패턴을 사용해 생성 과정을 캡슐화할 수 있다. (변경 시에도 호환성 유지)
2 생성 과정을 팩터리 클래스로 추출해 재사용할 수 있다.
3 복잡한 생성 과정을 캡슐화하고, 객체 생성 방법을 명확히 구분할 수 있다.
4 생성과 사용을 분리해 복잡한 코드를 간결하게 만들 수 있다.
✅ "생성 과정이 복잡하거나, 상황에 따라 다른 객체를 만들어야 할 때, 팩터리 패턴을 사용하라."
