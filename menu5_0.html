Observer Pattern

"객체 사이에 일대다(one-to-many) 의존성을 정의하여, 어떤 객체의 상태가 변할 때 그 객체에 의존하는 모든 객체가 자동으로 통지(알림)되고 갱신되도록 한다."

설명
주제(Subject) 객체의 상태가 변하면
옵저버(Observer) 들이 자동으로 알림을 받고 스스로 갱신(update) 한다는 구조이다.
그래서 주제와 옵저버 사이의 느슨한 결합(loose coupling) 을 이룰 수 있다.

🔵 옵저버 패턴을 사용하는 이유

이유 설명
객체 간 결합도 감소 주제(Subject)와 옵저버(Observer)가 느슨하게 연결되기 때문에, 서로 직접 참조하거나 강하게 의존하지 않는다. (코드를 수정하기 쉬움)
변화에 유연하게 대응 주제의 상태가 바뀔 때, 여러 옵저버들이 자동으로 반응할 수 있어 확장성과 유지보수가 쉬워진다.
다수 객체에게 일괄 알림 하나의 주제가 여러 옵저버에게 동시에 상태 변경을 알릴 수 있어, 대규모 시스템에서 유용하다.
OCP 원칙 준수 새로운 옵저버를 추가해도 주제 코드를 변경할 필요가 없다. (확장에는 열려있고 변경에는 닫혀있음)

🟠 옵저버 패턴의 활용 사례

활용 분야 구체적 설명
GUI 이벤트 시스템 버튼 클릭 시 여러 이벤트 리스너들이 동작한다. (ex: Java Swing, JavaScript 이벤트 리스너).
뉴스 발행 시스템 뉴스 발행 주제(Subject)가 새 기사를 업로드하면 구독자(Observer)들에게 알림을 보낸다.
주식 거래 시스템 주식 가격(Subject)이 변하면 투자자 애플리케이션(Observer)들이 자동으로 가격을 갱신한다.
소셜 미디어 알림 누군가 글을 올리거나 좋아요를 누르면, 팔로워들이 알림을 받는다.
파일 시스템 모니터링 파일이나 폴더의 변경사항을 감시하는 프로그램에서, 파일 상태가 변경되면 알림을 받는다.
게임 개발 캐릭터의 상태(Subject)가 변하면 UI 표시나 다른 시스템(Observer)이 즉시 반응한다.
