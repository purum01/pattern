Command Pattern

"요청을 객체로 캡슐화하여, 서로 다른 요청, 큐잉, 로그 요청 등을 지원하고, 작업의 실행을 나중에 할 수 있게 한다."

설명
어떤 요청(명령, 작업)을 하나의 객체로 만든다.
요청을 저장하거나,
요청을 취소하거나,
요청을 다시 실행하거나,
요청을 큐(Queue)에 쌓아두었다가 실행하는 것도 가능해진다.

🔵 커맨드 패턴을 사용하는 이유

이유 설명
요청을 캡슐화 요청(명령, 작업)을 객체로 만들기 때문에, 실행 시점과 구현 방식을 숨길 수 있다.
요청과 수신자 분리 명령을 보내는 쪽(Invoker)과 실제 작업을 수행하는 쪽(Receiver)을 분리해서 코드 의존성을 줄인다.
요청의 저장 및 실행 제어 명령을 객체로 저장할 수 있어, Undo/Redo(실행 취소/재실행)나, 큐잉(Queue에 쌓기)이 가능해진다.
복잡한 연산 관리 여러 작업을 하나의 명령으로 묶거나, 작업 실행을 나중으로 미루는 등의 처리가 가능하다.

🟠 커맨드 패턴의 대표적인 활용 사례

활용 분야 구체적 설명
Undo/Redo 기능 사용자가 했던 행동(명령)을 저장해두고, 실행 취소(Undo)나 다시 실행(Redo)할 때 사용한다.
매크로 명령(Macro Commands) 여러 개의 명령을 하나로 묶어서 한 번에 실행할 때 (예: 복잡한 초기 설정)
GUI 버튼 처리 버튼 클릭 → 명령 실행 구조. 버튼은 명령 객체만 알고, 실제 수행은 몰라도 된다.
작업 큐 및 요청 지연(Queueing and Scheduling) 명령을 큐에 넣고, 순서대로 실행하거나, 일정 조건이 되면 실행하도록 설정할 수 있다.
트랜잭션 처리 여러 명령을 모아서 하나의 트랜잭션처럼 실행하거나, 실패 시 전체 취소할 때 사용한다.
